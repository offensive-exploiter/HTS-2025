Before you start — team & identity

1. Create your team

Teams must have exactly 2 members. Choose a professional-sounding team name (e.g., "BlueShield Labs", "NullPointerCrew"). Team names are final when you submit.

  • Confirm both members' full names
  • Confirm both members' contact emails
  • Designate a team leader for communication
2. Identity file (REQUIRED)

Create a single file named TEAMNAME_identity.txt or TEAMNAME_identity.png that clearly lists:

  • Team name (exact)
  • Member 1 — Full name & email
  • Member 2 — Full name & email
  • Team leader contact (email)

Example content for TEAM_ALPHA_identity.txt:

Team: TEAM_ALPHA
Member 1: Alice Example — alice@example.com
Member 2: Bob Example — bob@example.com
Team leader: alice@example.com
3. What to submit

You can submit either:

  • Consolidated submission (recommended): one PDF (or ZIP) that includes
    • Identity file
    • Screenshots / evidence showing captured flags
    • Short report (1–2 pages) describing approaches and tools used
    • Links to GitHub repo (if applicable) and demo video (optional)
  • Per-round submissions: submit flags after each round ends (end of day). Each must include the identity file.
4. Proof on LinkedIn

To earn LinkedIn points, post at least one daily update on LinkedIn with:

  • A short description of what you worked on
  • @BlackXmask tag and the hashtag #HTS2025
  • Screenshots or short clips as evidence (if applicable)

Include a link to your LinkedIn post in the submission report (optional but recommended).

5. File naming & examples (recommended)

Use consistent filenames so judges can quickly identify your materials:

  • TEAMNAME_identity.txt
  • TEAMNAME_round1_evidence.pdf
  • TEAMNAME_round2_evidence.zip
  • TEAMNAME_project_README.pdf
6. Submission channels

We do not provide a direct file uploader on this page. If you need to upload files to GitHub, use one of the following recommended options:

Project repo Create an issue (attach files) Create a Release (upload ZIP) Create a branch / push files

Recommended GitHub flow:

  • Create a ZIP or PDF containing your identity file + evidence and upload it to a new Release (recommended), or attach it to a new Issue. Alternatively, push a branch with your submission files and open a pull request.
  • When using Issues, attach files (supported up to GitHub limits). Include TEAMNAME in the issue title and the identity file in the attachments.
  • If using Releases, create a release titled HTS2025 - TEAMNAME and attach your ZIP/PDF.
NOTE: GitHub has file size limits. If your files exceed limits, provide a downloadable link (Google Drive/Dropbox) in the Issue or Release notes and set access appropriately.

Other submission options: Devpost upload or email to blackxmask.official@gmail.com.

7. Judging & verification

Judges will verify flags and evidence. If we need clarifications, we will contact the team leader.

  • Provide raw screenshots and short explanations of where flags were found
  • Include commands used (optional) to reproduce steps

Round-specific guidance

Round 1 — Folder Forensics / Cryptography (16 Nov)

Locate 9 flags across a large dataset. Provide screenshots and short notes showing how each flag was found. Recommended tools: strings, binwalk, exiftool, steghide.

  • Evidence: screenshots of extraction commands and the flag string (or a highlighted image)
  • If using scripts, include the script or command history in the consolidated ZIP

Round 2 — Reverse & Recover (17 Nov)

Round 2 will run on 17 Nov. This round focuses on reversing safe challenge binaries to extract flags and understanding program behavior. Read this section carefully — Round 2 requires clear, reproducible evidence.

What we expect

  • Recovered flags from reversing tasks with step-by-step evidence
  • Short notes explaining the approach for each flag
  • Reproducible commands, key screenshots from the decompiler/debugger, and any small helper scripts

Recommended tools

Use well-known reversing tools — include tool names and versions in your report:

  • Ghidra (decompiler)
  • IDAr Pro / IDA Free (if available)
  • x64dbg, x32dbg
  • radare2 / Cutter
  • strings, hexdump for quick checks

Evidence checklist for Round 2

  • One screenshot showing the decompiled function where the flag is derived (highlighted)
  • One screenshot from the debugger showing the flag being printed or memory content
  • If you used a script to extract or decode data, include the script and a short README for it
  • List the exact command lines used to open the binary, run the debugger, or export strings
Tip: Name your files like TEAMNAME_round2_flag1_screenshot.png so judges can quickly match evidence to flags.

Round 3 — Build a Project (18–19 Nov)

Round 3 asks teams to build a small cybersecurity project. Submit demo video (optional), README, source code or a packaged deliverable. Provide clear setup instructions and a short report explaining design choices.

  • Deliverables: README, source code (or binary), short demo video or screenshots.
  • Documentation: setup steps, dependencies, and a short design / threat model note.
  • Packaging: prefer a ZIP or a GitHub repo link; include a release or tag for easy access.
Round 3 heading styled for emphasis — deliverables should be well-documented and reproducible.

Submission checklist (one-page)

  • IDENTITY file: TEAMNAME_identity.txt/png (required)
  • Evidence: consolidated PDF or per-round evidence files
  • Round 2: decompiler screenshots + debugger screenshots + scripts (if any)
  • Report: short summary (1–2 pages)
  • LinkedIn proof links (if applicable)
  • Contact email for team leader

FAQ / Common pitfalls

  • Q: Can I submit flags captured after the round end? A: Yes — you can submit at the end of the round day. Consolidated submissions until 20 Nov are accepted.
  • Q: Do I need to include raw binaries? A: Not required; screenshots and commands are sufficient. If you include binaries, make sure they are safe and documented.
  • Q: What if my evidence is too large? A: Use a ZIP or a Google Drive / Dropbox link and include access instructions.
  • Q: What if a judge asks for clarification? A: Respond promptly to the team leader email so we can verify your work.
